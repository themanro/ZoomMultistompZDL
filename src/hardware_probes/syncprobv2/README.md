# SyncProbe v2 — bypass state[7]

SyncProbe v1 and v1.5 confirmed `state[24]` is reachable from
custom-handler context but produced silent audio. The reason —
documented in [build/ABI.md §5.2](../../../build/ABI.md) under
"state[7] → c00cc8c8 → the audio↔UI postbox at 0x11f03b00" — is that
the LineSel-cloned handler still flows the callback's return value
through:

1. `SHL.S1 A4, 0x16, A4` (left-shift by 22 — kills large integers),
2. `state[7]` tail-call, which calls `c00cc94c` then `c00cc8c8`,
   sanitizes the value (`SUBSP/SPDP/ABSDP/CMPEQSP` + a tiny double
   constant clamp), and finally posts an IPC message in the UI
   postbox at `0x11f03b00..0x11f03b1f` to update the displayed
   knob position.

That chain expects post-SHL knob values in the `< ~0.14` raw range.
Any larger uint32 lands as ~zero, which is what the silent v1/v1.5
results showed.

## v2's change

Patch the same LineSel-cloned knob1 handler twice. Both patches live
in [build/syncprobe_v2_handler.bin](../../../build/syncprobe_v2_handler.bin).

| blob offset | original instruction                              | v2 patch |
|-------------|---------------------------------------------------|---------|
| `+0x64`     | `LDW.D1T2 *+A7[31],B31` (read knob via state[31]) | `LDW.D1T2 *+A7[24],B31` — read state[24] BPM helper |
| `+0x80`     | `LDW *+A7[21],B31`                                | `STW.D1T1 A4,*+A0[5]` — **write A4 directly into params[5]** |
| `+0x84..8b` | SHL + MVK setup                                   | NOPs |
| `+0x8c`     | `LDW *++B15[2],B3` (pop)                          | `LDW *++B15[2],B3` (kept — pop B3) |
| `+0x90..ab` | state[7] tail-call + delay-slot setup             | `BNOP.S2 B3,5` + NOPs — return |

So `state[24]`'s raw uint32 return lands in `params[5]` with no
normalization, no SHL, no state[7] postbox sync. The displayed knob
position will be wrong (we don't update the UI postbox) — that's
acceptable for a diagnostic probe.

## Audio body

The DSP reads `params[5]` as a raw `uint32_t`, takes the low 8 bits,
and maps them to a 0..2 linear gain:

```c
unsigned int raw = paramsRaw[SYNCPROBV2_SYNC_SLOT];
float gain = (float)(raw & 0xFFu) * (2.0f / 255.0f);
```

The 8-bit cut lets us see *any* variation in the uint32 as a
continuous gain change. If `state[24]` returns `0`, gain is `0` and
output is silent. If it returns a non-zero value that changes with
tap tempo, the gain pulses or drifts audibly.

## Flash and test plan

| step | action | expected if v2 works |
|------|--------|----------------------|
| 1 | Boot pedal, browse to Filter, select `SyncPrV2` | shows up, opens without freeze |
| 2 | Unbypass | audio passes (gain may be silent or quiet — that's data) |
| 3 | Turn `Sync` knob | **note whether the audio changes** — see "Interpretation" below |
| 4 | Left-knob click to tap tempo at a steady ~120 BPM | audio gain pulses / drifts on each tap |
| 5 | Tap a faster BPM (~180), then slower (~80) | audio response shifts |
| 6 | Switch effect away and back | reload behaves the same way |

## Interpretation

* **Step 2 silent + step 3 silent + step 4 silent**: `state[24]` is
  returning 0 with the LineSel knob_id constant `B4=2`. Even with
  state[7] bypassed, the value is still 0. v2 confirms the call
  protocol (not just the post-processing) is wrong. Next move: try
  v2 variants with different `B4` constants the same way v1.5 swept
  them.
* **Step 2/3 has constant audio, no change on step 4**: `state[24]`
  returns a non-zero constant. Tap tempo doesn't influence it under
  our call shape. Suggests the call needs more setup (the
  `state[31](B4=4)` prelude TAPEECH3 does, or a specific `A4` value).
* **Step 4 produces audible variation**: `state[24]` is BPM-aware
  with our call shape. v2's bypass strategy works; v3 can refine
  the byte-pattern interpretation.
* **Any freeze**: the bypass `STW.D1T1 A4,*+A0[5]` plus skipping the
  postbox is itself unsafe. Less likely than the above — `params[5]`
  is a regular float slot the audio function already reads — but
  possible.
